<objective>
Update PropertyDetail to show project linkage (banner + link back to project) for units belonging to a project, and update EditProperty to lock project-level fields when editing a unit under a project.
</objective>

<must_haves>
- PropertyDetail shows "Part of: {ProjectName}" banner for units with projectId
- Banner links to `#project/{projectId}` 
- EditProperty makes project-level fields read-only (locked) when editing a unit with projectId
- Locked fields show visual lock indicator
- Editing standalone properties remains unchanged
</must_haves>

<tasks>

<task id="03.1" title="Add project linkage banner to PropertyDetail">
<read_first>
- src/pages/PropertyDetail.js
- src/backend/projectService.js
</read_first>
<action>
In PropertyDetail.js:

1. Import `getProjectById` from `../backend/projectService.js`

2. After fetching the property data, check if `item.projectId` exists. If so, fetch the project:
```javascript
let project = null;
if (item.projectId) {
  try { project = await getProjectById(item.projectId); } catch (e) { /* project may be deleted */ }
}
```

3. If project exists, render a banner at the top of the detail view (right after the back button, before the media slider):
```html
<a href="#project/${item.projectId}" class="project-context-banner" style="text-decoration:none;">
  <svg width="16" height="16" fill="none" stroke="currentColor" viewBox="0 0 24 24">
    <path stroke-linecap="round" stroke-linejoin="round" stroke-width="2" d="M19 21V5a2 2 0 00-2-2H7a2 2 0 00-2 2v16m14 0h2m-2 0h-5m-9 0H3m2 0h5M9 7h1m-1 4h1m4-4h1m-1 4h1m-5 10v-5a1 1 0 011-1h2a1 1 0 011 1v5m-4 0h4"></path>
  </svg>
  <span>Part of: <strong>${project.name}</strong></span>
  <svg width="12" height="12" fill="none" stroke="currentColor" viewBox="0 0 24 24" style="margin-left:auto">
    <path stroke-linecap="round" stroke-linejoin="round" stroke-width="2" d="M9 5l7 7-7 7"></path>
  </svg>
</a>
```

4. This banner should be clickable and navigate to the project detail view.
</action>
<acceptance_criteria>
- PropertyDetail for a unit with `projectId` shows "Part of: {ProjectName}" banner
- Banner is clickable and navigates to `#project/{projectId}`
- PropertyDetail for standalone properties (no projectId) shows no banner — unchanged behavior
- Banner uses the `.project-context-banner` CSS class from Plan 02
</acceptance_criteria>
</task>

<task id="03.2" title="Lock project-level fields in EditProperty">
<read_first>
- src/pages/EditProperty.js
- src/config/propertyFields.js
</read_first>
<action>
In EditProperty.js:

1. Import `PROJECT_FIELD_NAMES` from `../config/propertyFields.js` and `getProjectById` from `../backend/projectService.js`

2. After loading the property data for editing, check if `item.projectId` exists. If so, fetch the project.

3. When rendering form fields, for each field check if `PROJECT_FIELD_NAMES.has(field.name)` AND the item has a `projectId`. If both true:
   - Add the `field-locked` CSS class to the form-group wrapper div
   - Set the input/select to `readonly` or `disabled`
   - Pre-fill the value from the project data (not the unit data, since project is source of truth)

4. Add a small info banner at the top of the edit form when editing a project-linked unit:
```html
<div class="project-context-banner">
  <svg>...</svg>
  <span>Fields inherited from <strong>${project.name}</strong> are locked. 
  <a href="#project/${item.projectId}">Edit project</a> to change them.</span>
</div>
```

5. On form submission, strip any project-level field names from the update payload (don't allow overwriting project data through unit edit). The filter should use:
```javascript
const updateData = {};
Object.entries(formData).forEach(([key, value]) => {
  if (item.projectId && PROJECT_FIELD_NAMES.has(key)) return; // Skip locked fields
  updateData[key] = value;
});
```
</action>
<acceptance_criteria>
- EditProperty for a unit with `projectId` shows locked (grayed, non-editable) project-level fields
- Locked fields display project data, not unit data
- An info banner explains which fields are locked and links to project edit
- Form submission does NOT include project-level field values in the update payload
- EditProperty for standalone properties is completely unchanged
- The `field-locked` CSS class is applied to wrapped form groups of locked fields
</acceptance_criteria>
</task>
