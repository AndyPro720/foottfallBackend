# Phase 10.1: Project-Based Property Grouping & Unit Management - Context

**Gathered:** 2026-05-05
**Status:** Ready for planning
**Source:** User-provided specification with PDF field breakdown

<domain>
## Phase Boundary

This phase introduces a "Projects" concept to the property management PWA. Currently, every property (unit) is a standalone Firestore document in the `inventory` collection with all fields stored flat. Many properties share a common building/project (e.g., Phoenix Mall with 20 units), causing users to re-enter building-level data repeatedly.

**This phase delivers:**
1. A new `projects` Firestore collection to store shared building-level data
2. A "Project View" in the Home page — clicking a project expands to show its common info and all units
3. A way to create projects from the Add section
4. A way to add units under a project (from project view or from the regular Add form with project auto-populate)
5. Field-level locking on project-inherited data to prevent accidental corruption
6. Backward compatibility — standalone properties (no project) continue to work as-is

</domain>

<decisions>
## Implementation Decisions

### Data Model — Project-Level Fields (shared across all units)
The following fields are LOCKED as project-level (from user's PDF):
- Property Name (building/project name)
- Vicinity Brands
- Building Type
- Entry to Building Photo
- Google Map Link
- Exact Location / Address
- City
- Trade Areas
- Contact Details (contactName, contactDesignation, contactInfo)
- Building Facade (photo/video)
- Age of Building

### Data Model — Unit-Level Fields (individual per unit)
The following fields remain unit-specific:
- Carpet Area (size)
- Floor
- Price per sqft
- CAM
- Price Negotiability
- Frontage
- Mergeable?
- Mezzanine (toggle + size + clear heights)
- Total Clear Height
- Connected Load
- Whole Facilities section (parking, outside space, service entry, lift, BOH, fire exit, OC file)
- Whole Photos/Videos/Docs section EXCEPT building facade (unit facade, interior, signage, floor plan, CAD, presentation)

### UX Flow — Creating a Project
- In the "Add" section, there should be an option at the top (e.g., a toggle/tab) to switch between "Property" (existing behavior) and "Project"
- Project creation form captures only project-level fields
- Saved to new `projects` collection in Firestore

### UX Flow — Adding Units Under a Project
- **From Project View (Home):** User opens a project, sees all units below, and clicks an "Add Unit" button. The form auto-populates all project-level fields (locked/non-editable). User only fills unit-specific fields.
- **From regular Add form:** If user selects a project from a dropdown/multiselect, all project-level fields auto-populate and become locked. User fills remaining unit-specific data.

### UX Flow — Viewing Projects on Home Page
- Projects appear in the Home property list (possibly with a distinct card style/badge)
- Clicking a project opens a "Project Detail" view showing:
  - Common project info at the top
  - All units listed below (similar to current property cards)
  - "Add Unit" action button
- Clicking a unit opens the standard PropertyDetail view

### Field Locking / Protection
- When viewing/editing a unit that belongs to a project, project-level fields should be visually locked (read-only or behind a lock icon)
- This prevents accidental modification of shared data from unit level
- To edit project-level data, user must edit the project itself

### Backward Compatibility
- Existing standalone properties (no projectId) continue to work unchanged
- The Home page shows both projects and standalone properties
- The existing IntakeForm remains the default for standalone properties

### Agent's Discretion
- Exact visual design of project cards vs standalone property cards
- Navigation patterns (hash routing for project view)
- Whether to use subcollections or a flat reference (projectId field on inventory docs)
- Firestore security rules for the new `projects` collection
- How to handle project deletion (cascade vs orphan units)
- Mobile-first responsive design details

</decisions>

## Specific Ideas

- Auto-populate should fill AND lock fields when a project is selected
- Project cards on Home page should visually distinguish from standalone properties
- Building Facade photo belongs to the project level, but unit-level photos (unit facade, interior, etc.) stay with the unit
